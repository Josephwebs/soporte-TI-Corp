# TicketManager API - Decisiones de Arquitectura

Este documento resume las principales decisiones técnicas y los *trade-offs* asumidos durante la construcción del Backend en .NET 8, priorizando un diseño escalable sin incurrir en sobreingeniería.

- **Arquitectura Limpia (Clean Architecture):** Opté por estructurar la solución separando las responsabilidades de forma estricta (`Domain`, `Application`, `Infrastructure` y `API`). Si bien consideré el uso de librerías como `MediatR` para implementar el patrón CQRS, decidí inyectar los servicios directamente para esta fase. Esto mantiene el código claro y facilita su revisión técnica, preservando la separación de dominios sin añadir complejidad innecesaria.
- **Entity Framework Core (In-Memory vs SQL Server):** Para cumplir con los requerimientos prácticos y facilitar la evaluación del proyecto, configuré la API para inicializarse con una base de datos In-Memory aprovisionada con datos semilla. Esto permite al evaluador probar los endpoints inmediatamente sin necesidad de configurar cadenas de conexión. Los scripts SQL reales para entornos productivos se adjuntan en el directorio `/db`.
- **Manejo de Excepciones Centralizado:** Con el objetivo de mantener los controladores limpios, implementé un `ExceptionMiddleware` global. Este componente intercepta los errores (400, 404, 500) y estandariza las respuestas siguiendo la convención RFC 7807 (ProblemDetails), garantizando consistencia en las respuestas hacia el cliente web.
- **Seguridad "Secure-by-Default":** Aunque el requerimiento permitía simulaciones simples (como un header `X-User`), decidí implementar autenticación real con tokens JWT y autorización basada en roles (`[Authorize(Roles="Supervisor")]`) para reflejar un entorno productivo realista. Para no entorpecer las pruebas de frontend, configuré un mecanismo de "auto-login" transparente en la capa cliente.
- **Aislamiento mediante DTOs:** Como regla arquitectónica, las Entidades del Dominio nunca se exponen directamente en los endpoints. Toda la comunicación externa fluye a través de Data Transfer Objects (DTOs) validados explícitamente, lo cual previene vulnerabilidades de inyección masiva (*Overposting*).
